# Возможности Owner / Admin в мобилке Smenka — аудит (2026-06-19)

> Контекст: планируется убрать режим **owner** из приложения. В приложении остаются **сотрудник** и **admin** (последний — для просмотра/редактирования самых частых базовых показателей). Полноценное управление owner+admin переезжает в веб-админку.
> Источник: аудит реального гейтинга в `lib/` (3 параллельные проверки).

## Ключевой вывод

**В приложении owner и admin почти идентичны.** Почти весь функционал закрыт общим гейтом `owner+admin` (`OrganizationDetailCubit.isAdminOrOwner = isOwner || member.role==admin || superAdmin`). Чистый эксклюзив очень маленький:

- **Только owner:** инвайт-код (просмотр + ротация) и удаление организации. Создатель org автоматически становится owner.
- **Только admin (не owner!):** мутации ставок (создать/изменить/удалить ставку). Owner видит ставки read-only.

То есть «убрать owner, оставить admin» на уровне UI — **малая работа** (две owner-only ветки), но требует backend-решения по инвайт-коду/удалению org и миграции текущих владельцев (см. ниже).

## Что умеет admin в приложении сейчас (всё это же видит owner)

Помечено: 🟢 базовое частое (просмотр/лёгкое) — кандидат остаться в приложении; 🔧 тяжёлое управление — кандидат уехать в web-админку.

|---|---|---|---|

| Участники организации | view, **delete** участника | `members` | 🟢 просмотр / 🔧 удаление |
| Деталь сотрудника | view | `member_detail` | 🟢 да |

## Эксклюзив owner (исчезает из приложения вместе с режимом)

| Возможность | Экран | Куда деть |
|---|---|---|
| Инвайт-код: просмотр + **ротация** | `_org_invite_section` | → web-админка (или оставить admin'у в app) |

| Создание организации (создатель = owner) | `_create_org_modal`, super_admin | → web-админка / super_admin |

## Механика ролей и что значит «убрать owner-режим»

- Роль владельца = `Organization.ownerId == currentUserId` (`isOwner`). Плюс enum `OrgMembershipRole {owner, admin, employee}` в `myRole`.
- **ADR-001 Owner != Member:** владелец **не числится участником** организации. У него `myRole='owner'`, системной member-роли нет.
- **Риск миграции (важно):** если просто убрать owner-режим, владелец остаётся с `myRole=null` → `canManage=false` → теряет ВСЁ (инвайт-код, удаление, все админ-функции). Нужно backend-решение: **либо** при удалении режима владелец становится `admin`-участником, **либо** инвайт/удаление/создание живут только в web-админке под owner-аккаунтом.
- Навигация (`main_router`, ADR трёхтабовая): 4 таба (Shift, History, Organizations, Profile) + 5-й SuperAdmin только для `UserRole.superAdmin`. Owner-режим — **не отдельный таб**, а две ветки внутри `organization_detail`. Поэтому вырезание в UI точечное.

## Решение (принято 2026-06-20)

Владелец работает **только через веб-админку** — owner-режим из приложения убираем полностью. Текущие владельцы НЕ мигрируются в admin-участники: они просто уходят в web (бэкенд должен корректно пускать owner в админку — вопрос трека admin/backend, не мобилки). Инвайт-код переезжает в админку — заведена задача `../docs/tasks/invite_code_admin/`. **Граница «частых показателей» отдана мне** — решение ниже.

### Что ОСТАЁТСЯ в приложении для admin (частые базовые показатели, в основном просмотр)
- Смены сотрудников `org_shifts` + деталь чужой смены (read-only) — view + фильтры.
- Статистика `org_stats` — view + фильтры по периоду.
- Зарплатный отчёт `payroll` — view + фильтры (просмотр «кому сколько»).
- Участники `members` — view списка + переход в деталь.
- Деталь сотрудника `member_detail` — **только просмотр** (роль, кастомная роль, overrides, ставки — read-only).

Итог: мобильный admin = «дашборд на ходу» (read-only аналитика по сотрудникам/сменам/деньгам). Никакого тяжёлого управления.

### Что УБИРАЕТСЯ из приложения (→ только веб-админка)
- 🗺️ **Экран карты / рабочие точки** `work_locations`, `add_edit_location`, виджет `flutter_map_widget`, инициализация Yandex MapKit (явно по решению — убрать; заодно снимает главный web-блокер).
- Настройки организации `org_settings` (геопроверка / автозавершение / лимиты пауз).
- Кастомные роли `roles` (CRUD).
- Шаблоны чек-листов `checklist_templates` + `checklist_template_detail` (CRUD).
- Правка ставок `rate_form` и любые мутации в `member_detail` (роль / кастомная роль / overrides / удаление участника).
- Owner-эксклюзив: инвайт-код, удаление организации, создание организации.

### Открытый нюанс (на потом)
Если admin'у в приложении всё же нужна одна частая правка «на ходу» — наиболее оправданный кандидат это **правка ставки сотрудника** (payroll — частая операция). Сейчас выношу в web; включить обратно — точечно, по запросу.

> Реализация этих изменений в мобилке (выпил owner-режима + карты + урезание admin до read-only) — отдельная mobile-задача/ветка, ещё не начата.
